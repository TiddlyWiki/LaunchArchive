Banned IPs plugin for Movable Type

I couldn't find one online, so I kludged a Movable Type plugin that allows you to use banned IP numbers inside MT templates. Why would anyone ever want to do this? Let's say you want to be merciless and not only disallow offending IPs from posting comments and Trackback pings, but also restrict access to other parts of your site. This plugin allows you to automate blacklist entries for an Apache .htaccess file like so:

<Directory /some_directory>
<MTBannedIPs>
  Deny from <$MTBannedIP$> # banned from <$MTBannedIPBlogName$>
</MTBannedIPs>
</Directory>

This should create an Apache directive block similar to this:

<Directory /some_directory>
  Deny from xx.xxx.xxx.xx # banned from My Blog
  Deny from xxx.xxx.xxx.xx # banned from My Blog
  Deny from xxx.xxx.xx.x # banned from My Blog
</Directory>

Alternatively, you can add an "all" attribute to the <MTBannedIPs> tag to get all banned IPs across all blogs in an MT installation:

<MTBannedIPs all="1">

The result:

<Directory /some_directory>
  Deny from xx.xxx.xxx.xx # banned from My Blog
  Deny from xxx.xxx.xxx.xx # banned from My Blog
  Deny from xxx.xxx.xx.x # banned from My Blog
  Deny from xxx.xxx.xxx.xx # banned from George's Blog
  Deny from xx.xxx.xx.xx # banned from George's Blog
  Deny from xxx.xxx.xx.x # banned from Larry's Blog
</Directory>

Or maybe you just want an internal report, or to feed these to a reverse DNS lookup script, or create a public Page Of Shame. Whatever, there're probably more ingenious uses for this.

banned_ips.zip (1.14KB). BTW, I wrote this with the 2.x API, so I have no idea if it works with MT 3.x.

August 02, 2004

DHTML: still hard, but easier than ever

My ex-KnowNow cohort Joyce Park takes Outpost to the woodshed over their "almost defiantly IE-only" implementation.

Cross-browser DHTML is actually easier than ever today, especially when you make the effort to constrain your definition of DHTML to its core: DHTML = HTML object model and CSS manipulation with JavaScript. The argument currently being used to evangelize CSS — that standards-based CSS designs allow you to support more browsers at less cost — can also be said of DHTML for the past several years (and even longer if you let Netscape 4 twist in the wind).

The disconnect happens when developers conflate DHTML with decidedly non-DHTML functionality. XMLHTTP? That's an interface to the networking layer. Client-side XSLT? Data binding? RPC? Now we're moving outside the scope of dynamic HTML, no? These are problems that are still being solved (in some cases over and over and over again ad nauseam) and perhaps the browser is not the most ideal environment in which to solve all of them. It seems that a lot of folks are willing to go through a lot of pain to create a client-side solution — or struggle with a broken, poorly understood native implementation — that might be more easily solved with a server application and a hidden IFRAME.

At the end of the day, I don't buy the argument that web development is still so difficult that we must pick and choose the browsers to support. I'm willing to give Oddpost the benefit of the doubt; they started out extremely small with limited resources, and they've been very upfront about their choice. In the darkest weeks of the dotcom bust this choice made sense, but that dog won't hunt today.

I really wonder what dark magick Oddpost is using under the hood. Is it, as Evan says, truly taking advantage of kick-ass IE-only functionality? Or did the Oddfolks simply choose the path of least resistance and gone too far to turn back? Anyway, their browser support is Yahoo's! concern now. The same company that recently overhauled much of My Yahoo! to be much more standards-compliant.
